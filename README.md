# LLM Evaluation on Bug Triage (Mozilla Bugzilla)

An end-to-end NLP / LLM-evaluation project on 13,019 public Mozilla Bugzilla
bug reports that answers three questions:

1. **What themes do bugs repeatedly report?** (unsupervised embedding + clustering)
2. **Can a new bug automatically find its duplicate?** (semantic dedup retrieval with
   4,987 hard-label pairs — hit@k / MRR)
3. **Are the structured fields extracted by an LLM trustworthy?** (self-correcting
   extraction agent + LLM-as-judge calibrated with Cohen's κ against human labels)

## Overview

**Theme discovery (product analytics)**
- `text-embedding-3-small` embeddings cached in **ChromaDB** (re-runs are free)
- **KMeans** with silhouette-based k selection on a 3,000-bug sample
- Cluster names generated by **GPT-4o** from representative summaries
- **Wilson 95% CI** on topic prevalence (normal approximation is inaccurate at tails)

**Dedup retrieval (semantic search headline)**
- 4,987 `RESOLVED DUPLICATE` pairs as ground truth
- Cosine-similarity nearest-neighbour retrieval over the ChromaDB vector store
- Metrics: **hit@1 / hit@5 / hit@10 / MRR**

**Extraction agent (LLM evaluation headline)**
- LangGraph multi-agent graph: `supervisor` (LLM router) → `researcher` (ReAct tool
  loop) / `extractor` (JSON output) / `critic` (schema + faithfulness check)
- Three retrieval tools used by the researcher: `search_similar_bugs`,
  `read_bug`, `component_prior` — all deterministic (vector search / dict lookup,
  zero LLM cost)
- **MCP grounding** — field catalog (definitions + enums) served by `mcp_server.py`
  (stdio MCP server); single source of truth for extractor and critic prompts
- **LLM-as-judge** auto-rater calibrated with **Cohen's κ** on 50 human-labelled bugs

## Data

| File | Description |
|---|---|
| `bugs.csv` | 13,019 bugs: id, component, severity, summary, description, role |
| `dup_pairs.csv` | 4,987 `(dup_id, master_id)` hard-label pairs |
| `human_labels.csv` | Human-annotated area / severity / kind for the extraction eval sample |
| `desc_cache.json` | GPT-4o cluster-name cache (avoids repeated API calls) |

Data download script: `download_bugzilla.py`.

## Run

```bash
pip install -r requirements.txt
cp .env.example .env          # add OPENAI_API_KEY
python pipeline.py            # all 8 steps end-to-end
```

Steps 1–4 and Step 7 are free (local / cached).
Steps 5–6 call the OpenAI API (~300 calls for the 50-bug extraction eval).

## Walkthrough notebook

`analysis.ipynb` walks through all 8 steps interactively — one markdown
explanation + code + visualization per step:

```bash
jupyter lab analysis.ipynb    # or open in VS Code / Cursor
```

Set `SAMPLE = True` in the Setup cell for a fast smoke-test (~10 bugs, no API cost
beyond the embedding step).

## Structure

| File | Role |
|---|---|
| `pipeline.py` | All 8 steps as importable functions + `main()` entry point |
| `analysis.ipynb` | Interactive walkthrough; calls `pipeline.py` functions |
| `development.ipynb` | Exploratory scratch notebook the pipeline was distilled from |
| `mcp_server.py` | stdio MCP server — serves field catalog and component taxonomy |
| `download_bugzilla.py` | Downloads and caches bug data from Bugzilla REST API |
| `requirements.txt` | Python dependencies |

## Pipeline steps

| Step | What it does | Cost |
|---|---|---|
| 1 load | Load `bugs.csv` + `dup_pairs.csv`, build text field | free |
| 2 embed | Embed all bugs into ChromaDB (cached) | OpenAI embeddings (once) |
| 3 cluster | KMeans + silhouette k-selection + GPT-4o cluster naming | OpenAI chat (small) |
| 4 topic_stats | Wilson CI on topic prevalence, save chart | free |
| 5 extract | Extraction agent on 50 bugs | OpenAI chat (~300 calls) |
| 6 kappa | Cohen's κ vs. human labels, trajectory chart | free (uses step 5 output) |
| 7 dedup | hit@1 / hit@5 / hit@10 / MRR on 4,987 pairs | free (uses cached embeddings) |
| 8 readout | Print final summary | free |
